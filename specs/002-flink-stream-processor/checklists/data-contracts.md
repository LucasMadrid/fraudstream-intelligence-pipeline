## Type Safety & Schema Enforcement

- [x] CHK001 — Is `geo_network_class` defined as an Avro `enum` type rather than a free `string`? The doc lists valid values (RESIDENTIAL, BUSINESS, HOSTING, MOBILE, UNKNOWN), but a free string allows invalid values to be written without schema-level rejection, making downstream switch logic fragile. [Clarity, Completeness, Contract §geo_network_class] — PASS: `enriched-txn-v1.avsc §geo_network_class` type changed to `["null", {"type": "enum", "name": "NetworkClass", "symbols": ["RESIDENTIAL", "BUSINESS", "HOSTING", "MOBILE", "UNKNOWN"]}]`. Both contracts/ and pipelines/processing/schemas/ copies updated.
- [x] CHK002 — Is `error_type` in `processing-dlq-v1.avsc` defined as an Avro `enum` rather than a free `string`? Undocumented error category values break consumer triage logic; an enum enforces the closed set (SCHEMA_VALIDATION_ERROR, LATE_EVENT_BEYOND_ALLOWED_LATENESS, ENRICHMENT_EXCEPTION). [Clarity, Contract §error_type] — PASS: `processing-dlq-v1.avsc §error_type` type changed to Avro enum with symbols [SCHEMA_VALIDATION_ERROR, LATE_EVENT_BEYOND_ALLOWED_LATENESS, ENRICHMENT_EXCEPTION]. Both contracts/ and pipelines/processing/schemas/ copies updated.
- [x] CHK003 — Are the decimal `precision` (18) and `scale` (4) of `vel_amount_*` fields in `enriched-txn-v1.avsc` consistent with the `amount` field precision/scale in the upstream `txn_api_v1` schema? Precision mismatch between input and aggregated output causes silent rounding in accumulated totals. [Consistency, Gap, Contract §vel_amount_1m] — PASS: `amount` field uses `precision: 18, scale: 4`; all four `vel_amount_*` fields use identical `precision: 18, scale: 4` — no rounding mismatch.

## Schema Evolution & Registry

- [x] CHK004 — Is the Avro schema compatibility mode (BACKWARD, FORWARD, or FULL) documented and registered in Schema Registry for both `txn.enriched` and `txn.processing.dlq`? Without a compatibility mode, breaking field changes reach consumers silently and without a validation gate. [Completeness, Gap, Contract §schema_version] — PASS: Decision: **BACKWARD**. Rationale: the scoring engine (consumer) is the downstream authority — producers (Flink processor) must never ship a schema that breaks an existing consumer. BACKWARD enforces this: consumers can be upgraded independently; only additive changes (new optional fields with defaults) are permitted without a topic rename. Documented in plan.md §Security Requirements §Schema Registry Authentication: "Both `txn.enriched` and `txn.processing.dlq` subjects MUST be registered with BACKWARD compatibility mode before the first production deployment."
- [x] CHK005 — Is the `schema_version` field documented as a topic-level evolution signal (e.g., triggers a `txn.enriched.v2` topic rename), and is this strategy consistent with the registered Schema Registry compatibility mode? A string field with default "1" does not enforce versioning; the upgrade path must be defined before v2 work begins. [Consistency, Ambiguity, Contract §schema_version] — PASS: Logged as **TD-004** in spec.md §Technical Debt. The gap is formally accepted: the v2 upgrade path (topic-rename vs. subject-version, consumer migration procedure, schema_version increment mechanism) must be defined before any v2 work begins. Blockers for v1 are none; TD-004 gates future v2 planning.
- [x] CHK006 — Is the Avro namespace (`com.fraudstream.processing.enrichment`) documented as stable, and is there a requirement that namespace changes are treated as breaking changes for any consumer that uses the fully-qualified record name for schema resolution? [Completeness, Gap, Contract §namespace] — PASS: Logged as **TD-005** in spec.md §Technical Debt. The gap is formally accepted: namespace `com.fraudstream.processing.enrichment` is implicitly stable for v1; any future namespace change must be treated as a breaking change requiring a Schema Registry subject rename and consumer migration. TD-005 gates any refactor that touches the namespace.

## Field Provenance & Semantics

- [x] CHK007 — Is the provenance of `geo_lat` and `geo_lon` in `enriched-txn-v1.avsc` documented unambiguously? The field doc says "from caller — forwarded unchanged", but the enrichment pipeline derives geolocation from `caller_ip_subnet` via GeoLite2. Are these coordinates from the raw request (forwarded) or from the GeoLite2 lookup (derived)? [Clarity, Ambiguity, Contract §geo_lat, §geo_lon] — PASS: `enriched-txn-v1.avsc §geo_lat/§geo_lon` docs explicitly state "Geographic latitude from the raw API request — forwarded unchanged from txn.api. NOT derived from GeoLite2". GeoLite2 derives `geo_network_class`, `geo_country_code`, `geo_city`; coordinates come directly from the caller. Both contracts/ and pipelines/processing/schemas/ copies updated.
- [x] CHK008 — Is the `enrichment_latency_ms` calculation (`enrichment_time - processing_time`) documented as measuring processor-internal latency, distinct from the end-to-end latency in SC-001? The two boundaries differ: `processing_time` is set by the ingestion producer, not the Flink consumer receipt — using it as a latency start point underreports enrichment latency. [Clarity, Consistency, Contract §enrichment_latency_ms, Spec §FR-012, Spec §SC-001] — PASS: `enriched-txn-v1.avsc §enrichment_latency_ms` doc now explicitly states: "MEASUREMENT BOUNDARY: processing_time is set by the ingestion producer (feature 001) at API receipt — NOT at Flink consumer receipt. This field measures total pipeline latency including Kafka queuing time — NOT pure Flink-internal latency." Both schema copies updated.
- [x] CHK009 — Is `device_known_fraud`'s "always null in v1" state documented as a contract-level semantic — specifically, that consumers must treat `null` as "not yet evaluated" rather than `false`? Without this distinction, consumers that coerce null-to-false make an incorrect fraud signal interpretation. [Clarity, Contract §device_known_fraud, Spec §TD-002] — PASS: `enriched-txn-v1.avsc §device_known_fraud` doc corrected to: "NOT YET WIRED IN V1 — always null. Consumers MUST treat null as 'not evaluated' (TD-002). DO NOT coerce null to false — a false interpretation signals no fraud signal was available, not that the device is known-clean." Both schema copies updated.
- [x] CHK010 — Is the `masking_lib_version` and `processor_version` field format (semver, commit hash, custom string?) documented with an example, and is there a requirement that the format is stable across releases? Consumers that parse these for debugging may break on format changes. [Clarity, Contract §masking_lib_version, §processor_version] — PASS: `enriched-txn-v1.avsc §processor_version` doc specifies "Format: '{feature-branch}@{semver}' (e.g. '002-stream-processor@1.0.0'). Populated at job startup from the PROCESSOR_VERSION environment variable." `§masking_lib_version` doc specifies "Format: semver string (e.g. '1.2.3'). Forwarded unchanged from the ingestion producer's masking library." Both schema copies updated.

## Consumer Contract Completeness

- [x] CHK011 — Are the `vel_amount_*` fields documented as per-currency aggregates (TD-001: no cross-currency normalization in v1), and is there a requirement that the enriched schema carries the `currency` field so consumers can interpret amounts correctly? The `currency` field is present, but no doc explicitly links it to the multi-currency limitation. [Completeness, Traceability, Contract §vel_amount_1m, Spec §TD-001] — PASS: `enriched-txn-v1.avsc §vel_amount_*` docs updated to include: "NO cross-currency normalisation (TD-001: deferred to v2). Consumers MUST filter by the currency field before comparing velocity amounts across accounts or time windows." Both schema copies updated.
- [x] CHK012 — Are the DLQ reprocessing fields (`source_topic`, `source_partition`, `source_offset`, `original_payload_bytes`) documented as sufficient for deterministic replay — including confirmation that the Confluent Schema Registry magic byte + schema ID prefix is preserved in `original_payload_bytes`? Without the schema ID, replay requires out-of-band schema resolution. [Completeness, Contract §original_payload_bytes] — PASS: processing-dlq-v1.avsc `original_payload_bytes` doc explicitly states "including Confluent Schema Registry magic byte + schema ID prefix"; all four reprocessing fields present.
- [x] CHK013 — Is the relationship between `transaction_id` in `enriched-txn-v1.avsc` and `transaction_id` in `processing-dlq-v1.avsc` documented as a stable join key for reconciliation — i.e., that the same UUID appears in exactly one of the two topics for every input event? [Traceability, Contract §transaction_id] — PASS: Both `enriched-txn-v1.avsc §transaction_id` and `processing-dlq-v1.avsc §transaction_id` docs include: "RECONCILIATION CONTRACT: every input event from txn.api appears on exactly one of txn.enriched (this topic) or txn.processing.dlq — never both, never neither. Use transaction_id as the join key for reconciliation." DLQ nullable case documented: null only when Avro deserialization of the original payload fails entirely. Both schema copies updated.
